## 1. 現在の仕様サマリー

### アプリ概要
- **名称:** 比率計算アプリ
- **形態:** 単一HTMLファイルのPWA（Service Worker + manifest.json）
- **データ保存:** localStorage（4つのキー: recipes, folders, tags, settings）
- **対象デバイス:** モバイルファースト（max-width: 480px）、デスクトップ対応

### 画面構成（4画面 + モーダル群）
1. **レシピ一覧画面 (screen-list):** フォルダ/レシピのリスト表示、検索、タグフィルタリング
2. **レシピフォーム (modal-recipe-form):** ボトムシートモーダルでの新規登録/編集
3. **計算画面 (screen-calc):** 総量入力からの按分計算 + 材料基準の逆算
4. **設定画面 (screen-settings):** 端数処理モード、タグ管理、エクスポート/インポート

### 機能一覧

#### レシピ管理
- レシピの新規作成（名前、材料名:比率のペア）
- レシピの編集、削除、複製
- フォルダ分類（レシピをフォルダに所属させる）
- タグ付け（複数タグ対応、フォルダスコープ設定可能）
- ドラッグ&ドロップによる並べ替え（デスクトップ + タッチ対応）
- 長押しによる一括編集モード（一括削除、一括移動、一括タグ編集）
- レシピ名/材料名でのテキスト検索
- タグによるフィルタリング（OR/AND切り替え可能）

#### フォルダ管理
- フォルダの作成、名前変更、削除、複製
- フォルダ削除時の選択肢（レシピも削除 / レシピを未所属に移動）
- フォルダ内レシピ表示（フォルダビュー）
- フォルダの並べ替え

#### タグ管理
- タグの作成（名前、カラー、スコープ設定）
- 24色プリセット + HSLカスタムカラーピッカー
- タグのスコープ（全フォルダ / 特定フォルダのみ）
- タグの並べ替え（フィルタバーでドラッグ）
- タグの一括削除（設定画面から長押しで発動）

#### 計算機能
- 希望の総量を入力 → 比率に基づいて各材料の量を按分計算
- 5つの端数処理モード: 四捨五入、切り捨て、切り上げ、近似値、端数処理なし
- 「材料から計算」: 任意の材料に数値を入力 → 他の材料を比率で自動算出
- 計算欄の追加（複数セット同時計算可能）
- +10/+50/+100/リセットのクイック入力ボタン
- よく使う数値の保存/呼び出し
- 計算結果のクリップボードコピー
- 計算結果のメモへの自動出力（タイムスタンプ付き）
- レシピごとのメモ（自動保存）

#### 設定
- 端数処理モード（5種類）
- 最後に開いたレシピの記憶（ON/OFF）
- データのエクスポート（JSON、項目選択可能）
- デ���タのインポート（JSONファイル読み込み）
- 設定のリセット

#### PWA対応
- Service Worker（キャッシュファースト戦略）
- manifest.json（standalone表示、ポートレート固定）
- アイコン（192x192, 512x512）

## 2. バグ・問題点リスト

### 重大度: HIGH（機能破損・データ損失リスク）

#### BUG-001: isRecipeFormDirty() が編集時に常にtrueを返す
- **場所:** L4291-4301
- **内容:** `if (currentRecipeId) return true;` により、既存レシピの編集画面を開くと、何も変更していなくても「編集内容を破棄しますか？」の確認ダイアログが必ず表示される。
- **影響:** ユーザーが編集画面を開いて確認だけして閉じたい場合に、毎回不要な確認が出る。

#### BUG-002: タグ作成時にorderが設定されない
- **場所:** L3956-3964 (`saveTagFromModal`)
- **内容:** 新規タグ作成時に `order` プロパティが設定されない。`loadData` のマイグレーション (L1876) で index ベースの order が付与されるが、リロードするまで order が undefined のままになる。
- **影響:** リロード前のタグ並び順が不安定になる可能性がある。

#### BUG-003: レシピ保存後にcurrentFolderViewがnullにリセットされる
- **場所:** L4546 (`saveRecipe`)
- **内容:** レシピ保存後に `currentFolderView = null` が実行され、フォルダビューから編集→保存すると必ずメインリストに戻される。
- **影響:** フォルダ内でレシピを編集して保存した後、フォルダビューに戻れない。ユーザーの作業フローが中断される。

#### BUG-004: 複製レシピのtagIdsが参照共有のリスク
- **場所:** L3022-3029 (`duplicateRecipe`)
- **内容:** `...src` でスプレッドしているが、`ingredients` と `tagIds` は浅いコピーのため、配列の参照が共有される可能性がある（ただし `ingredients` の各要素はプリミティブなので実質問題なし、`tagIds` は文字列配列なので実害は低い）。
- **影響:** 低。ただし `ingredients` 内のオブジェクトを変更すると元レシピにも影響する。

### 重大度: MEDIUM（機能上の不具合・UX劣化）

#### BUG-005: フォルダコンテキストメニュー関数が未使用（デッドコード兼）
- **場所:** L3557-3596 (`showFolderContextMenu`)
- **内容:** この関数はどこからも呼び出されていない。フォルダのメニューボタンは `openFolderSheet` を呼んでいる。ただし、関数内で `confirm()` を使った古い削除ロジックがあり、フォルダ削除モーダル (`modal-folder-delete`) と二重で存在している。
- **影響:** コードの混乱要因。古いロジックが残存。

#### BUG-006: historyStackのscreen名に 'folder-view' が入るが実際のscreenは 'list'
- **場所:** L1904-1914
- **内容:** `showScreen('folder-view')` では内部的に `screen-list` を表示し、historyStackには 'folder-view' が積まれる。これ自体は動くが、`folder-view` の画面遷移状態が screen DOM と一致しないため、戻るボタンの挙動が複雑になっている。
- **影響:** 低。動作はするが、将来のメンテナンスリスク。

#### BUG-007: escAttr関数のエスケープが不完全
- **場所:** L5132
- **内容:** `escAttr` は `"` と `'` のみエスケープするが、`&`, `<`, `>` はエスケープしない。`escHtml` とは別関数として存在するが、attribute コンテキストではすべてエスケープすべき。
- **影響:** 材料名やレシピ名に `&` や `<` を含む場合、HTML属性内で不正な出力になる可能性がある。

#### BUG-008: 単位(unit)入力変更時にupdateAllCalcIngredientSetsが呼ばれるが計算結果には影響しない
- **場所:** L4862-4863
- **内容:** `input-unit` の input イベントで `calculate()` と `updateAllCalcIngredientSets()` を両方呼んでいるが、単位は表示用であり、材料ベースの計算結果には影響しない。不要な再計算。
- **影響:** 低。パフォーマンスへの影響は微小。

#### BUG-009: 近似値モードでk=0になる可能性
- **場所:** L4911
- **内容:** `const kLow = Math.max(1, Math.floor(kApprox))` で最低値を1に制限しているが、非常に大きなintSumに対して小さいtotalInputを入力した場合、k=1固定になり希望量と大幅に異なる結果が出る。ユーザーへの説明が不足。
- **影響:** 中。ユーザーが混乱する可能性。

### 重大度: LOW（軽微な問題）

#### BUG-010: タグ並べ替え時にtouchstartがpassive:trueで登録されている
- **場所:** L2232-2234
- **内容:** タグソートチップの touchstart が passive:true で登録されているため、`e.preventDefault()` が呼べない。ただしtouchstart内では呼んでいないので実害なし。

- **影響:** デッドステート変数。

## 4. 設計上の問題

### DESIGN-001: 単一HTMLファイルに全コード（約5340行）
- **内容:** CSS(約1430行) + HTML(約270行) + JavaScript(約3640行) が1ファイルに集約されている。
- **影響:** メンテナンス性が低い。ただしPWAとしてシンプルなデプロイを実現しているメリットもある。
- **推奨:** 現時点では分割の必要性は低いが、機能追加が続く場合は検討すべき。

### DESIGN-002: グローバルステートの多用
- **内容:** 30以上のグローバル変数（recipes, folders, tags, settings, currentRecipeId, calcRecipeId, bulkEditMode, etc.）でアプリ状態を管理している。
- **影響:** 状態間の依存関係が暗黙的で、バグの温床になりやすい。特に `currentFolderView`, `historyStack`, `bulkEditMode` の相互作用が複雑。

### DESIGN-003: DOM直接操作によるUI更新
- **内容:** 仮想DOMやテンプレートエンジンなし。すべて `innerHTML` と `createElement` で手動構築。
- **影響:** 再レンダリング時にDOM全体を破棄→再構築するパターンが多く（`renderRecipeList`, `renderTagFilterBar` など）、パフォーマンスとメモリリークのリスクがある（イベントリスナーの蓄積）。

### DESIGN-004: navigator.clipboard.writeText のエラーハンドリング不足
- **場所:** L4997-4999, L5017-5019 他
- **内容:** clipboard API の `.then()` のみで `.catch()` がない。HTTPS以外の環境や権限不足時にサイレントに失敗する。
- **推奨:** try-catch またはフォールバック実装。

### DESIGN-005: confirm() / alert() の使用
- **場所:** 複数箇所（レシピ削除 L2966、フォルダ削除 L3577、比率正規化 L4440、インポート L4079-4086 他）
- **内容:** ネイティブの confirm/alert を使用しているが、他の削除操作（一括削除）ではカスタムモーダルを使っている。UIの一貫性がない。
- **影響:** UXの不統一。ネイティブダイアログはモバイルブラウザで見た目が異なる。

### DESIGN-006: レシピ保存時のsavedAmountsフィールドの扱い
- **場所:** L4537-4543 (`saveRecipe`)
- **内容:** 新規レシピ作成時に `savedAmounts` が含まれないが、`memo` は含まれる。既存レシピ編集時は `...recipes[idx]` のスプレッドで `savedAmounts` が保持されるが、明示的でない。
- **推奨:** 新規作成時に `savedAmounts: []` を明示的に設定。

### DESIGN-007: Service Workerのキャッシュ戦略
- **内容:** Cache-First 戦略のみ。ネットワーク更新のフォールバックなし。アプリ更新時にユーザーが古いキャッシュを使い続ける可能性。
- **推奨:** Stale-While-Revalidate または更新通知の仕組み。

### DESIGN-008: genId() の衝突可能性
- **場所:** L5124
- **内容:** `Date.now().toString(36) + Math.random().toString(36).slice(2, 7)` で生成。理論上、同一ミリ秒で複数呼び出すと衝突する可能性がある（実用上はほぼ問題ない）。
- **影響:** 極めて低い。
